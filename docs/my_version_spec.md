# 畢業專題：AI 智慧會議室管理系統 (個人獨立開發版規格書)

## 1. 系統架構與技術選型
本系統為單一檔案的輕量化原型，旨在完整實現從「預約、防呆、轉譯、AI摘要到自動通知」的完整自動化流程：
* **前端/後端介面**：使用 Streamlit 快速構建網頁 UI。
* **資料儲存**：採用在地端輕量化的 `meeting_database.json` 檔案。
* **核心 AI 技術**：整合 OpenAI Whisper API (語音轉文字) 與 Google GenAI Gemini API (結構化摘要)。
* **自動化通知**：利用 Python 內建的 `smtplib` 對接 Gmail SMTP 伺服器。

---

## 2. 核心功能模組與實作規格

### 模組一：使用者與權限管理 (RBAC)
* **測試資料定義**：系統內置 5 筆固定員工資料（包含身分證字號、生日、Email、部門與角色）。
* **登入驗證**：使用者輸入「身分證字號（帳號）」與「生日（密碼）」進行登入。
* **權限查閱邏輯**：
    * 若角色為 `Boss`：可調閱並翻看系統內「所有部門」的會議紀錄。
    * 若角色為 `Employee`：僅能查閱「自己有參與（身分證字號在與會名單內）」的會議紀錄。

### 模組二：會議預約與前置防呆 (Validation)
當使用者在 Streamlit 網頁點擊「確認預約」時，系統必須在後端執行以下嚴格的防呆驗證：
1.  **時間範圍限制**：會議的 `start_time` 與 `end_time` 必須落在 **09:00 至 21:00** 之間。
2.  **時間間距限制**：預約時間的分鐘數必須剛好為 `00` 或 `30` 分（以 30 分鐘為單位）。
3.  **1 小時前限制（預約/變更/取消）**：會議的 `start_time` 減去 `當前系統時間` 必須 $\ge 1 \text{ 小時}$。
4.  **通知觸發**：預約成功後，系統立即使用 `smtplib` 發送純文字信件給與會人員：
    > 「[與會人姓名]，您於 X 月 X 日 X 點 X 分有一場關於 [會議主題] 的會議，發起人為 [發起人姓名]。」

### 模組三：語音轉文字 (STT) 與 Gemini 結構化摘要
* **音檔處理**：提供 Streamlit 檔案上傳元件，支援上傳 `.mp3` 或 `.wav` 會議錄音檔。
* **文字轉譯**：呼叫 Whisper API 將音檔完整辨識為文字。
* **Gemini 強型別摘要**：使用 Pydantic 模型（`GeminiMeetingReport`）強行鎖定 Gemini 的輸出格式。Prompt 必須要求 AI 準確產出以下四項內容：
    1.  `meeting_title`: 根據內容自動擬定的正式會議主題。
    2.  `executive_summary`: 核心決策與會議背景的高階主管摘要。
    3.  `key_topics`: 會議討論的主要議題清單 (`list[str]`)。
    4.  `action_items`: 明確的分工待辦清單，含責任人與期限 (`list[str]`)。

### 模組四：半小時自動發信追蹤機制
* **確認機制**：Gemini 生成摘要後，系統狀態設為 `is_confirmed = False`。
* **路徑 A (手動)**：發起人在網頁看見摘要，半小時內點擊「確認發送」，系統立刻發送摘要信並將狀態改為 `True`。
* **路徑 B (自動)**：系統內部使用執行緒（`threading.Timer`）進行 30 分鐘倒數。若倒數結束發起人仍未確認，系統將自動調用發信模組，強制將摘要 Email 寄給所有出席人員。

## 3. 精簡版 Python 代碼開發指引 (精簡控制在 200 行內)
為了避免多餘編碼並保持代碼好維護，你的個人版 `app.py` 請遵循以下精簡原則：
* **不使用複雜外部資料庫**：一律讀寫 `meeting_database.json`。
* **合併 UI 與邏輯**：直接在 Streamlit 的 `st.button` 點擊事件中撰寫資料寫入與防呆判斷，省去繁瑣的 MVC 架構。
* **原生排程**：直接用 `threading.Thread` 或 `time.time()` 做簡單的時間差計算來處理 30 分鐘的未確認通知，不要引入外部複雜的 Celery 或排程套件。
